## Summary

Remove the largest current design drift before the main integration task: the executable still runs the old monolithic prototype path (`src/main.rs` + `src/api.rs`) with external `ffplay`/`mpv`, hard-coded colors, raw station structs, and fake visualizer state.

This issue should make the active binary path honest and non-conflicting before `MIK-010` wires the replacement modules into the real event loop.

## Context

Design sources:

- `docs/SPEC.md` — native Rust playback, theme-driven UI, responsive layout, MVP scope
- `docs/implementation-guidelines.md` — module boundaries, typed domain boundaries, no old shotgun parsing
- `docs/ui-design-decisions.md` — Theme-owned colors, Search Console / Split Mini direction
- `docs/superpowers/plans/2026-06-27-radio-replacement.md` — Task 10 remains the real integration task
- Drift audit after MIK-001..009 found that replacement modules are aligned, but `src/main.rs` / `src/api.rs` still expose the pre-replacement app path.

## Scope

Do a narrow cleanup/quarantine, not full integration:

- Remove or clearly isolate the old `ffplay`/`mpv` playback path from the primary binary.
- Remove or clearly isolate the old `src/api.rs` top-vote API path from the primary binary.
- Remove hard-coded old-prototype Ratatui UI from the primary binary path.
- Keep the crate compiling and tests passing.
- Keep `audio_spike` as the documented manual native-audio verification binary.
- Leave real terminal event loop, key handling, debounce, persistence, auto-play, and native runtime wiring to `MIK-010`.

A minimal temporary `main` is acceptable if it is explicit that replacement integration is handled by `MIK-010`; it should not pretend to be the final player.

## Acceptance Criteria

- Running the primary binary no longer launches external players (`ffplay`/`mpv`).
- `src/main.rs` no longer contains the old monolithic prototype app state, hard-coded theme colors, fake visualizer, or raw Radio Browser station model.
- `src/api.rs` is removed, unused, or explicitly quarantined so it is not part of the active app path.
- The active binary path does not contradict the documented native-audio/replacement architecture.
- No `utils`, `helpers`, `common`, or `misc` catch-all modules are introduced.
- `MIK-010` remains responsible for the actual integrated TUI/event-loop/audio/search/persistence behavior.

## Validation

Run:

```bash
cargo fmt --check

```

Also scan:

```bash
rg -n "ffplay|mpv|Command::new|Color::" src/main.rs src/api.rs
```

Expected: no matches in the primary binary/prototype path, except if a deliberately archived/quarantined file is documented as non-active.
