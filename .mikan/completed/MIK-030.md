## Summary

Fix focus handling so station/result navigation only happens when the Results/Stations pane is focused. Pressing Up/Down or Space while focus is in another pane should not move the Results pane cursor.

## User Report

When focus is outside the Results pane, pressing Up/Down or Space still moves the Results pane cursor. This makes non-Results panes feel unsafe to interact with because global navigation leaks into station selection.

## Expected Behavior

- Up/Down and `j`/`k` affect only the currently focused navigable pane.
- Space toggles playback only when that action is valid for the focused pane, or otherwise does nothing.
- Results/Stations selection changes only while Results/Stations is focused.
- Existing global controls such as `/`, `Tab`, theme/volume keys, and quit continue to work as designed.

## Scope

- Audit key dispatch in the TUI event loop and app reducer for focus-aware behavior.
- Ensure Browse/Sections, Search/input, Now Playing, or other non-Results focus states do not accidentally dispatch station selection actions.
- Add regression tests covering Up/Down and Space from at least one non-Results focus state.
- Coordinate with MIK-020 if it is implemented in the same keyboard-focus slice.

## Acceptance Criteria

- Pressing Up/Down outside Results does not move the Results cursor.
- Pressing Space outside Results does not move or unexpectedly toggle the selected station.
- Pressing Up/Down inside Results keeps the current station navigation behavior.
- Regression tests cover focus-specific key handling without requiring a real terminal.
