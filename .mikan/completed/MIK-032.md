## Summary

Make Spectrum-family visualizers more polished by interpolating theme spectrum colors across low/mid/high instead of using hard band cutoffs.

## Context

Current `Theme::spectrum_color(position)` maps normalized band position to one of three discrete colors: `spectrum_low`, `spectrum_mid`, or `spectrum_high`. Spectrum-family visualizers such as SpectrumStack, PeakDots, and SkylinePeaks already call this method per rendered column, so making this method interpolate colors will turn the existing low/mid/high triplets into smooth gradients while keeping all palette ownership centralized in `theme.rs`.

## Scope

- Update spectrum coloring so normalized positions smoothly interpolate low → mid → high.
- Keep all color math and palette knowledge inside `theme.rs`; do not add ad hoc colors in `ui.rs`.
- Preserve exact endpoint behavior: position `0.0` resolves to `spectrum_low`, midpoint resolves to `spectrum_mid`, and position `1.0` resolves to `spectrum_high`.
- Handle non-RGB / named terminal colors safely by converting existing palette colors to RGB equivalents inside `theme.rs`.
- Keep existing visualizer renderers and layout unchanged.
- Update docs/comments only if they still describe hard low/mid/high bands.

## Acceptance Criteria

- `Theme::spectrum_color` returns a smooth interpolated color between low/mid/high for RGB-capable theme palettes.
- Low, mid, and high endpoints remain stable for every built-in theme.
- Non-RGB named colors used by existing themes still produce deterministic, readable interpolated colors.
- Spectrum-family UI tests still pass and at least one test proves an intermediate column is not merely the old discrete low/mid/high cutoff.
- No visualizer-specific color constants are introduced outside `theme.rs`.

## Validation

```bash
cargo fmt --check
cargo test theme
cargo test ui
cargo check
```
