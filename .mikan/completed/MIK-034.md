## Summary

Keep the Browse `Favorites` source as the single favorites mode, showing all saved favorites. Do not add a separate current-context/filtered favorites mode.

## Context

The user initially considered two modes, but revised the product direction after review: the default and only favorites behavior should be simple. `Favorites` means all saved favorites, not only favorites from the current list/search.

## Selected UX

- Browse has one `Favorites` entry.
- `Favorites` shows every saved favorite in persisted insertion order.
- Do not expose `All Favorites` or `Current Favorites` labels.
- Do not add a toggle or dedicated key for current-context favorites.

## Scope

- Remove/revert any `Current Favorites` source, state, tests, and docs added during the earlier MIK-034 attempt.
- Keep or restore existing all-favorites behavior.
- Ensure user-facing text says `Favorites`, not `All Favorites`.
- Removing a favorite while in Favorites should update the visible list immediately and clamp selection.
- Empty Favorites should show the existing helpful all-favorites empty state.
- Update README/spec/design docs so they describe the single-mode `Favorites` behavior.

## Acceptance Criteria

- Browse exposes exactly one favorites entry: `Favorites`.
- `Favorites` shows all persisted favorites in saved insertion order.
- No `Current Favorites` / `All Favorites` user-facing labels remain.
- No current-context favorites state or filtering behavior remains.
- Favorite removal updates Favorites immediately and clamps selection.
- App/UI tests cover the restored single-mode behavior and guard against reintroducing current-filtered labels.
