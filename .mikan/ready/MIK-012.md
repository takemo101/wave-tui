## Summary

Finish the MVP replacement by polishing offline/error behavior, removing obsolete prototype paths, updating docs, and running final verification.

## Context

Source docs:

- `docs/SPEC.md` — Offline / Network Failure, Success Criteria
- `AGENTS.md` — Verification and documentation rules

## Scope

- Make offline/network failure state visible in all layout tiers.
- Keep cached/favorite/built-in retry options available while offline.
- Remove or clearly isolate obsolete `ffplay`/`mpv` playback code from the main app path.
- Ensure `audio_spike` remains as a documented spike/manual verification tool, or archive it intentionally.
- Update README to reflect the actual replacement behavior.
- Update docs if implementation changed any durable design decision.

## Acceptance Criteria

- Online search failure shows a clear offline/error state without crashing.
- The app still allows retrying previous station, favorites, and built-in candidates.
- Broken stations are temporarily disabled for the session.
- External-player playback is no longer the primary app path.
- README controls and quick start match implemented behavior.
- `docs/SPEC.md`, `docs/ui-design-decisions.md`, `docs/audio-spike.md`, and implementation plan are updated if reality diverged.
- Final success criteria in `docs/SPEC.md` are checked manually or documented as remaining gaps.

## Validation

Run:

```bash
cargo fmt --check
cargo test
cargo check
cargo clippy -- -D warnings
```

Manual checklist:

- first launch starts silently with recommendations
- play a curated station
- FFT reacts to real audio
- quit/relaunch resumes previous station
- `--no-auto-play` starts silently
- search updates while typing
- favorite persists across restart
- theme cycles and persists
- offline banner appears during network failure
- wide/medium/compact layouts do not overlap
